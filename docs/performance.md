# Performance

## 优化背景

商品列表、Dashboard、选品分析、商品详情和候选池页面都依赖后端接口数据。在线上 Serverless 部署、远程数据库读取或网络较慢时，如果页面必须等待接口完成后再展示内容，用户会长时间看到 Loading 状态。

当前优化目标是在不改变接口行为和业务逻辑的前提下，改善二次访问、缓存命中和接口较慢场景下的页面体感速度。这套优化面向商品池、候选池、Dashboard 和 AI 报告等通用能力，不局限于当前手机支架种子品类。

## 性能问题

优化前的主要问题：

- 页面首屏内容依赖接口返回。
- Serverless 冷启动或远程 Supabase 查询较慢时，页面会长时间空白等待。
- 商品列表筛选如果频繁请求接口，会增加等待时间和后端压力。
- 多个页面读取同一商品池时，可能重复访问 Supabase。

## 方案设计

当前版本使用前后端组合优化。

### 前端缓存

前端对以下数据使用 `localStorage` 缓存：

- 完整商品池
- Dashboard 数据
- 商品详情
- 候选池数据

缓存有效期为 6 小时。

页面采用 stale-while-revalidate 体验：

1. 如果存在可用缓存，先展示缓存数据。
2. 同时在后台请求最新接口数据。
3. 新数据返回后更新页面和缓存。
4. 如果缓存不可用，则回退到正常 Loading 和接口请求流程。

### 前端二次筛选

商品列表在完整商品池缓存可用时，尽量在前端完成关键词、类目、利润率、排序和策略筛选，减少重复接口请求。

### 后端短期缓存

后端对 Supabase 商品池读取增加 5 分钟内存缓存，用于减少 Dashboard、商品列表、详情页等重复读取 Supabase 的开销。

## 测量口径

测量工具：Playwright + Microsoft Edge Headless。

测量口径：

- 从 `page.goto()` 开始计时。
- 到页面首个有意义内容可见结束计时。
- Products 以 `.product-card` 可见为结束点。
- Dashboard 以 `.dashboard-page__hero` 可见为结束点。
- Analysis 以 `.analysis-card` 可见为结束点。
- ProductDetail 以 `.detail-page__hero` 可见为结束点。
- Favorites 以 `.favorite-card` 可见为结束点。

## 本地 warm API 测量结果

测试环境：`http://127.0.0.1:5176` + `http://localhost:3002`，同一 BrowserContext。

| 页面 | 无前端缓存 | 缓存命中 | 节省时间 | 提升比例 |
| --- | ---: | ---: | ---: | ---: |
| Dashboard | 1609ms | 353ms | 1256ms | 78% |
| Products | 443ms | 380ms | 63ms | 14% |
| Analysis | 439ms | 278ms | 161ms | 37% |
| ProductDetail | 490ms | 266ms | 224ms | 46% |
| Favorites | 1052ms | 271ms | 781ms | 74% |

## 模拟慢接口测量结果

测试方式：使用 Playwright route 将 `/api/**` 请求人为延迟 4000ms，模拟线上 Serverless 冷启动或远程数据库请求较慢的场景。

| 页面 | 无前端缓存 | 缓存命中 | 节省时间 | 提升比例 |
| --- | ---: | ---: | ---: | ---: |
| Dashboard | 4697ms | 350ms | 4347ms | 93% |
| Products | 4621ms | 379ms | 4242ms | 92% |
| Analysis | 4648ms | 278ms | 4370ms | 94% |
| ProductDetail | 4643ms | 293ms | 4350ms | 94% |
| Favorites | 4614ms | 274ms | 4340ms | 94% |

## 结果解读

这次优化主要提升以下场景的体感速度：

- 二次访问。
- 缓存命中。
- 接口较慢。
- Serverless 冷启动或远程数据库响应慢。

在模拟 4 秒接口延迟时，主要页面首个有意义内容可见时间从约 4.6 秒降到约 0.27-0.38 秒，说明 stale-while-revalidate 对慢接口场景有明显改善。

## 方案限制

- 首次访问仍然取决于后端接口、网络和部署平台冷启动。
- `localStorage` 缓存适合轻量数据，不适合无限增长的商品池。
- 缓存数据可能短时间落后于后端最新数据。
- 后端内存缓存受 Serverless 实例生命周期影响，不保证跨实例共享。
- 当后续接入 1688 和 Amazon API 后，需要重新评估缓存粒度、刷新策略和数据新鲜度要求。

## 后续优化方向

- 为真实 API 接入后的数据刷新设计更明确的 TTL 策略。
- 增加接口耗时和缓存命中率观测。
- 对商品池做分页或增量更新，避免数据规模扩大后一次性加载过多。
- 为候选池和 AI 上下文设计更细粒度的缓存失效策略。
- 在部署环境中补充端到端性能回归检查。
