# Repository Maintenance Guide

## 项目名称

极瑞AI跨境选品分析平台

Jirui AI Cross-border Product Selection Platform

## 项目范围

本项目是一个面向跨境轻小件选品场景的 AI 决策辅助平台。系统围绕商品数据分析、利润评估、竞争强度判断、物流与风险识别、候选池管理、可视化看板和 AI 选品建议构建。

当前版本以手机支架作为种子品类和演示数据，通过 Supabase 商品数据、JSON 备份数据、Express API、React 页面和 AI Provider 后端封装，验证跨境轻小件选品分析流程。平台定位不局限于手机支架，后续会围绕 1688 API、Amazon API、多轻小件品类扩展、选品规模扩大和数据时效性提升继续迭代。

## 维护原则

1. 只处理用户当前明确要求的任务。
2. 不擅自扩大需求范围。
3. 不为了小任务做大范围重构。
4. 优先保持现有功能、接口和数据结构稳定。
5. 文档整理任务不得修改业务代码、接口行为或依赖版本。
6. 如果确实需要改动业务逻辑，必须单独说明原因、影响范围和验证方式。
7. 所有公开文档应面向产品、技术、维护和使用场景，不写个人职业材料。
8. 品类扩展应优先采用数据驱动或配置化方式，不应把某一个种子品类写死为长期唯一业务范围。

## 公开文档规则

公开 GitHub 文档应体现这是一个长期维护的产品型项目。适合公开保留的内容包括：

- 产品介绍
- 技术架构
- API 说明
- 部署说明
- 性能优化
- 路线图
- 更新日志
- 技术决策记录
- 数据模型说明
- 安全与维护规范

公开文档建议结构：

```text
README.md
docs/
  product-overview.md
  architecture.md
  api.md
  deployment.md
  performance.md
  roadmap.md
  changelog.md
  DATA_MODEL.md
```

实际新增文档应按项目需要最小化处理，不为了目录完整而创建空泛文档。

## 私有文档规则

以下内容不应放在公开文档目录中：

- 个人记录
- 草稿记录
- 面向个人职业规划的材料
- 本地提示词
- 未脱敏的调试记录
- 含有真实账号、密钥、内部链接或部署细节的记录

建议放入本地私有目录：

```text
_private/
private-notes/
local-drafts/
career-notes/
*.local.md
```

公开 README 和 docs 不应链接 `_private/` 目录。若需要将私有记录转为公开文档，必须重新整理为产品化、技术化、脱敏后的说明。

## 安全规则

1. 不提交 `.env`、`server/.env`、`client/.env`。
2. 不在公开文档中写入真实 API Key、数据库密钥、service role key 或 token。
3. `SUPABASE_SERVICE_ROLE_KEY` 只能配置在后端环境变量中。
4. `ZHIPU_API_KEY`、`NVIDIA_API_KEY` 等第三方 AI Key 只能配置在后端环境变量中。
5. 前端只允许使用公开配置，例如 `VITE_API_BASE_URL`。
6. `server/.env.example` 只能包含变量名和占位符。
7. 如果发现真实密钥曾被提交，应先提示风险，并建议轮换密钥与清理 Git 历史。
8. 回答或文档中不要打印完整密钥值。

## 代码改动边界

文档整理任务中不要改动：

- `client/src/` 业务逻辑
- `server/routes/` 接口行为
- `server/services/` AI 调用逻辑
- `server/utils/` 业务计算逻辑
- `package.json` 和锁文件
- 数据库表结构和迁移逻辑

除非用户明确要求，否则不要主动添加：

- 登录注册
- JWT 鉴权
- 权限系统
- 支付功能
- 大规模状态管理重构
- Docker 化部署
- 真实爬虫
- 与跨境轻小件选品无关的业务模块

## 当前种子品类与产品迭代方向

当前版本以手机支架作为种子品类，用于验证商品池、候选池、利润评估、风险识别、推荐评分、Dashboard、AI 问答和 AI 商品报告等核心流程。后续主要围绕真实数据源和规模化选品能力扩展：

- 接入 1688 API，提升货源、采购成本、供应商与发货信息的时效性。
- 接入 Amazon API，提升候选商品、销量、评价、评论和竞争数据的更新能力。
- 从手机支架种子品类扩展到更多跨境轻小件候选商品。
- 强化 AI 建议与结构化指标之间的一致性和可解释性。
- 完善部署、监控、缓存、异常兜底和数据刷新策略。

## 文档变更检查清单

文档变更完成后，应检查：

- 公开文档是否仍含个人职业包装语气。
- README 是否能说明项目是什么、解决什么问题、如何运行、如何维护。
- docs 是否围绕产品、架构、数据、性能、路线图展开。
- 手机支架是否被表述为当前种子品类，而不是平台长期唯一业务范围。
- 私有文档是否没有被公开链接。
- `.gitignore` 是否覆盖 `_private/`、`*.local.md` 和其他本地私有材料目录。
- 公开文档是否只出现环境变量名和占位符，没有真实密钥。
- Git 状态中是否只包含预期文档变更。

## 验证建议

常用只读检查：

```bash
git status --short
git ls-files server/.env client/.env .env
```

公开文档关键词检查：

```bash
rg -n "个人职业材料|包装语气|未脱敏|旧平台名|旧英文名" README.md AGENTS.md docs
```

敏感变量检查应只确认变量名和占位符，不输出真实密钥值。
