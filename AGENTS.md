# 智能体工程协作指南

本文件用于约束智能体和后续维护者在本仓库中的开发、文档、数据、安全和验证行为。处理任何任务时，应优先遵守本文件中的项目边界与操作规则。

## 项目上下文

项目名称：

- 极瑞AI跨境选品分析平台
- Jirui AI Cross-border Product Selection Platform

项目方向：

本仓库是一个面向跨境轻小件选品场景的 AI 决策辅助平台。系统围绕商品数据分析、利润评估、竞争强度判断、物流与风险识别、候选池管理、可视化看板和 AI 选品建议构建。

当前样例数据以手机支架相关轻小件商品作为种子品类，用于验证选品分析流程。不要把项目描述为长期只服务手机支架品类。后续迭代方向包括接入 1688 API、Amazon API、扩展更多轻小件品类、扩大候选商品规模，并提升市场与供应链数据时效性。

## 核心协作原则

1. 只处理用户当前明确要求的任务。
2. 保持改动范围收敛，并与现有架构和代码风格一致。
3. 不为了小任务做大范围重构。
4. 除非任务明确要求，否则优先保持现有接口、数据结构和用户可见行为稳定。
5. 文档类任务不得修改业务逻辑、运行行为、依赖版本或数据库结构。
6. 如果确实需要改动业务逻辑，必须说明原因、影响范围和验证方式。
7. 品类扩展应优先采用数据驱动或配置化方式，不应把当前种子品类写死为长期业务边界。
8. 尊重工作区中已经存在的用户改动，不回滚无关变更。

## 文件安全与归档规则

本仓库中禁止执行任何文件删除命令，包括但不限于：

- `rm`
- `del`
- `Remove-Item`
- `git clean`
- 通过脚本或 shell alias 永久删除文件
- 清空回收站或执行等价的破坏性清理

当文件或目录需要从当前位置移除时，必须归档，不得删除。

归档根目录：

```text
D:\agent-archive\
```

归档规则：

1. 尽量保留原始相对路径和文件名。
2. 如果归档目标已经存在同名路径或文件，必须追加时间戳，不能覆盖。
3. 除非用户明确要求覆盖归档文件，否则不得覆盖已有归档内容。
4. 归档操作只能通过移动或复制到归档目录完成，不能通过删除实现。
5. 每次执行归档后，必须在回复中明确告知用户完整归档路径。

## 公开文档规范

公开文档应体现本仓库是一个长期维护的产品型和工程型项目。README 和 docs 可以包含：

- 产品介绍
- 技术架构
- API 行为
- 部署说明
- 性能优化
- 路线图
- 更新日志
- 技术决策记录
- 数据模型说明
- 安全与维护规范

推荐的公开文档结构：

```text
README.md
docs/
  product-overview.md
  architecture.md
  api.md
  deployment.md
  performance.md
  roadmap.md
  changelog.md
  DATA_MODEL.md
```

只创建确实能提供产品、技术或维护价值的文档。不要为了补齐目录结构创建空泛占位文档。

公开文档应使用平台名称“极瑞AI跨境选品分析平台”，项目方向统一为“轻小件跨境选品分析”。可以说明当前样例数据以手机支架相关轻小件为主，但必须明确这是当前种子品类，不是平台长期唯一业务范围。

## 私有材料处理规范

以下材料不得从公开 README 或 docs 中链接：

- 本地记录
- 草稿记录
- 非产品化说明
- 本地提示词
- 未脱敏调试记录
- 含有真实账号、密钥、内部链接或部署细节的记录

此类材料应放入本地私有路径，例如：

```text
_private/
private-notes/
local-drafts/
*.local.md
```

如果私有材料需要转为公开文档，必须先改写为产品化、技术化、脱敏后的说明，再移动或链接。

## 安全规则

1. 不提交 `.env`、`server/.env`、`client/.env`。
2. 不在公开文档中写入真实 API Key、数据库密钥、service role key 或 token。
3. `SUPABASE_SERVICE_ROLE_KEY` 只能存在于后端环境变量中。
4. `ZHIPU_API_KEY`、`NVIDIA_API_KEY` 等 AI Provider 密钥只能存在于后端环境变量中。
5. 前端只能使用公开配置，例如 `VITE_API_BASE_URL`。
6. `server/.env.example` 只能包含变量名和占位符。
7. 如果发现真实密钥可能曾被提交，应提示用户风险，并建议轮换密钥与清理 Git 历史。
8. 不要在回复、日志、文档或示例中打印完整密钥值。

## 代码改动边界

处理文档或维护类任务时，不要修改：

- `client/src/` 业务行为
- `server/routes/` API 行为
- `server/services/` AI Provider 或报告生成行为
- `server/utils/` 业务计算逻辑
- `package.json` 或锁文件
- 数据库结构或迁移逻辑

除非用户明确要求，否则不要主动添加：

- 登录注册
- JWT 鉴权
- 角色或权限系统
- 支付功能
- 大规模状态管理重构
- Docker 化部署
- 真实爬虫或抓取系统
- 与跨境轻小件选品无关的业务模块

## 产品迭代边界

当前种子品类：

- 手机支架相关轻小件商品

当前已验证的平台能力：

- 商品池与候选池
- 利润评估
- 风险识别

- 从当前种子品类扩展到更多跨境轻小件商品。
- 强化 AI 建议与结构化指标之间的一致性。
- 完善部署、监控、缓存、异常兜底和数据刷新策略。

## 验证检查清单

完成改动前，应根据任务风险和影响范围选择合适的检查。

常用只读检查：

```bash
git status --short
git ls-files server/.env client/.env .env _private
```


敏感配置检查：

- 确认公开文档只包含环境变量名和占位符。
- 检查过程中不要输出真实密钥值。

文档完成度检查：

- README 能说明项目是什么、解决什么问题、如何运行、如何维护。
- 公开 docs 围绕产品、架构、数据、性能、部署和路线图展开。
- 当前种子品类被表述为样例数据，而不是永久业务边界。
- 公开 README 和 docs 不链接私有路径。
- Git 状态中只包含本次任务预期变更。
